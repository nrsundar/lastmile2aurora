DDL Analysis Documentation

Try DDL Analysis

Overview

DDL Analysis automatically creates tables with sample data, analyzes query performance, and recommends optimal indexes for your PostgreSQL database.

What It Does:
  • Parses your table DDL statements
  • Creates temporary tables with 1,000 sample rows
  • Analyzes query execution plans
  • Recommends priority-based indexes
  • Generates production-ready DDL statements
Benefits:
  • Identifies missing indexes before production
  • Estimates performance improvements
  • Supports complex multi-table queries
  • Provides AWS Aurora optimizations
  • Downloadable index creation scripts

Step-by-Step Usage Guide

Step 1: Prepare Your DDL Statements

Gather the CREATE TABLE statements for all tables involved in your query. Include:

  • Primary key definitions
  • Foreign key relationships
  • Column data types
  • NOT NULL constraints
Step 2: Identify Your Target Query

Choose the SELECT query you want to optimize. Best candidates include:

  • Frequently executed queries
  • Queries with performance issues
  • Complex joins across multiple tables
  • Queries with ORDER BY or GROUP BY clauses
Step 3: Configure Analysis Settings
  • Schema Name: Unique identifier for your analysis
  • Source Type: Original database platform
  • Target Version: PostgreSQL version (14, 15, or 16)
Step 4: Run Analysis

The system will:

  1. Create temporary schema with your tables
  2. Generate realistic sample data
  3. Execute EXPLAIN ANALYZE on your query
  4. Identify optimization opportunities
  5. Test performance with recommended indexes
Step 5: Review Recommendations

Examine the results in priority order:

  • HIGH - Critical performance impact
  • MEDIUM - Moderate improvement
  • LOW - Minor optimization
Quick Tips
Pro Tip: Start with your slowest queries for maximum impact.
Note: Test recommended indexes in development before production.
AWS Aurora: Includes Aurora-specific optimizations and parallel query features.

Sample Input & Expected Output

Sample DDL Input:
CREATE TABLE customers (
    customer_id SERIAL PRIMARY KEY,
    first_name VARCHAR(50) NOT NULL,
    last_name VARCHAR(50) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    city VARCHAR(50),
    country VARCHAR(50),
    created_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE orders (
    order_id SERIAL PRIMARY KEY,
    customer_id INTEGER NOT NULL,
    order_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    total_amount DECIMAL(10,2),
    status VARCHAR(20),
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);
Sample Query Input:
SELECT c.first_name, c.last_name, o.total_amount
FROM customers c
INNER JOIN orders o ON c.customer_id = o.customer_id
WHERE c.city = 'New York'
    AND o.order_date > '2024-01-01'
    AND o.status = 'completed'
ORDER BY o.total_amount DESC
LIMIT 100;

Advanced Features & Best Practices

Supported Query Patterns:
  • Simple Filters: WHERE column = value
  • Range Queries: WHERE date > '2024-01-01'
  • Multi-table Joins: INNER, LEFT, RIGHT, FULL OUTER
  • Sorting Operations: ORDER BY single/multiple columns
  • Grouping: GROUP BY with aggregations
  • Subqueries: Correlated and non-correlated
Index Types Recommended:
  • BTREE: Default for most use cases
  • Composite: Multi-column indexes for complex filters
  • Partial: Conditional indexes for specific scenarios
  • Unique: For constraint enforcement
Best Practices:
Implementation Order:
  1. Start with HIGH priority indexes
  2. Test one index at a time
  3. Measure performance impact
  4. Monitor index usage with pg_stat_user_indexes
Considerations:
  • Indexes speed reads but slow writes
  • Monitor storage usage growth
  • Regular VACUUM and ANALYZE
  • Consider composite vs. multiple single indexes
AWS Aurora Specific:
  • Parallel query optimizations
  • Read replica considerations
  • Auto-scaling impact on indexing

Troubleshooting Common Issues

Common Errors & Solutions:
Error: "DDL content is required"

Solution: Ensure you've pasted valid CREATE TABLE statements in the DDL field.

Error: "Table creation failed"

Solution: Check DDL syntax. Remove database-specific features not supported in PostgreSQL.

Error: "Sample data generation error"

Solution: Verify foreign key relationships are correctly defined in your DDL.

Error: "Analysis timeout"

Solution: Simplify complex queries or reduce the number of tables for initial testing.

Performance Tips:
For Large Schemas:
  • Start with core tables only
  • Focus on most critical queries first
  • Use representative table subsets
Data Type Considerations:
  • Oracle NUMBER → PostgreSQL NUMERIC
  • SQL Server NVARCHAR → PostgreSQL VARCHAR
  • DATETIME → TIMESTAMP
  • Auto-converted during analysis
