The Problem

What SCT Does Well ✅

Schema migration — DDL, tables, indexes, constraints, views

What's Left Behind ❌

Application-layer SQL — DML queries embedded in code. Discovered broken or slow in production after cutover.

Common Issues

  • Oracle syntax fails on PG (NVL, ROWNUM, DECODE)
  • Queries fast on Oracle, slow on Aurora PG
  • Test data ≠ production volume → false comparisons

What LastMile2Aurora Does

🏷️
Tag & Track

Tag business-critical queries with SQL comments. Match across Oracle and PG by tag, not SQL hash.

📊
Compare

Oracle data blocks vs PG pages. Execution time, row counts, data correctness. Flag regressions.

Why Tags?

The Matching Problem

Same business query → different SQL syntax → different hash

-- Oracle
SELECT /* tag:order_lookup */
  o.order_id, NVL(o.amount, 0)
FROM orders o
WHERE ROWNUM <= 100

-- PostgreSQL
SELECT /* tag:order_lookup */
  o.order_id, COALESCE(o.amount, 0)
FROM orders o
LIMIT 100

Same Tag = Same Query

How It Works

  • Customer tags top 50-100 critical queries
  • Same tag in Oracle and PG code
  • Tool reads V$SQL + pg_stat_statements
  • Matches by tag, compares performance

Data Volume Warning

If Oracle has 10M rows and PG has 100K, plans will differ. That's not a regression — it's a test data issue. We flag this.
